01 / DEMO 简介

是什么 · 面向谁 · 能做什么

基于 Next.js 的校园健身互助预约网站，服务复旦健身社三类核心用户

📋

预约带练

学员选择场馆、时段、教练，填写训练需求并提交预约。支持每周额度、违约禁约、时段冲突三重校验，保障资源合理分配。

✅

教练审核

认证教练查看收到的预约，通过/拒绝审核，标记课程完成。教练可自主管理可约时段，12 小时内未审核自动取消。

🛡️

管理后台

管理员审核教练申请、发布公告、查询预约记录、管理黑名单（解除禁约），完整覆盖社团运营全场景。

核心业务规则（全部实现）

✓
每周 3 次预约上限 满额时按钮禁用并提示
✓
24 小时取消规则 开课前 24h 可免费取消,之后记违约
✓
违约 3 次禁约 30 天 自动触发禁约提示
✓
同教练同时段唯一 已约时段不可再次选择
✓
预约状态机完整 pending → approved → completed/cancelled
✓
新手 / 女生友好标签 降低入门门槛,保护隐私
02 / 创作思路

灵感来源 · 痛点 · 取舍

为什么用互助预约模式解决校园健身指导问题

💡 灵感来源

来自社团真实痛点

复旦健身社拥有大量优秀健身爱好者，但新社员缺乏科学训练指导，经常因动作不规范导致受伤；同时教练资源分散在微信群、QQ 群中，缺乏统一的预约工具,协调成本高、效率低。

🎯 想解决的问题

三大真实痛点

① 新社员零基础入门难，不知如何开始训练；
② 教练带练时间难协调，缺乏高效预约工具；
③ 违约成本低导致教练时间浪费，需要一套信用约束机制促进社区健康发展。

⚖️ 为什么做这个方向

判断与取舍

健身是大学生群体的高频需求,互助模式成本低、可扩展性强,适合社团运营。预约平台能提升训练效率、保障训练安全,同时通过信用机制促进社区健康发展。MVP 优先跑通核心流程,后续可扩展课程包、积分体系等。

04 / TRAE 实践过程

完整开发流程 · 9 阶段递进

从需求分析到上线部署,全程使用 TRAE 完成

P0

需求分析与方案设计

让 TRAE 读取《复旦健身社互助预约平台-开发交接文档》和 Design 高保真页面,自动生成项目文件结构、实现计划(P0-P9 九个阶段)、核心组件拆分、Mock 数据结构。确认 MVP 技术选型:Next.js 14 + React 18 + TypeScript + Tailwind CSS 3。
约 1 小时 · 方案对齐
P1

初始化与数据层

创建项目骨架,迁移设计 Token,安装依赖。实现 types.ts(全部 TS 类型定义)、constants.ts(业务规则常量)、utils.ts(工具函数)、mock-data.ts(日期动态化 Mock 数据)。TRAE 自动识别文档中的日期过期问题并动态化处理。
约 1 小时 · 数据层搭建
P2

全局骨架与首页

实现 AppContext(全局状态管理)、NavBar、Footer、Modal、状态组件、登录页。完成首页 10 个模块:Hero、校区概览、场馆、教练筛选、新手步骤、友好支持、器材、规则、公告、社团介绍。
约 1 小时 · 骨架与首页
P3-P4

核心页面开发

场馆介绍页(Tab 切换 + 校区筛选 + 3 可约场馆 + 江湾旗舰 + 其他校区)。教练预约页(3 步向导 + 时段网格 + 三重校验 + 防重复提交)。个人中心(用户卡 + 额度进度 + 违约警示 + 预约 Tab + 取消弹窗 24h 分叉)。
约 2 小时 · 核心流程
P5

教练中心与管理后台

教练中心(资料编辑 + 时段管理 + 收到预约审核 + 通过/拒绝/标记完成)。管理员后台(4 Tab:教练审核、公告发布、预约记录、黑名单管理)。完整覆盖社团运营场景。
约 1 小时 · 后台开发

联调打磨与部署

修复 hooks 规则错误、ESLint 配置优化。npm run build 生产构建验证通过(8 个路由全部静态生成)。Cloudflare Pages 部署生成永久公网链接。修复 Hydration 错误(Footer 动态导入)。
